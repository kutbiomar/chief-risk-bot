# ChiefRiskBot — Live UI/UX Audit
*Performed 2026-05-03 against `frontend-mvp/` served via `.claude/serve.js` on `:8000`, with the FastAPI backend on `:8001` (AUTH_MODE=local). Demo workspace: Whitmore Family Office.*

---

## 0. Persona reconciliation

The user-supplied persona list (CIO, Analyst, Admin, Family Office, UHNW Individual) is **not 1:1** with the canonical `admin/business/ChiefRiskBot_User_Jobs.md`, which defines four personas (Fund Manager, Family Office head, UHNW Individual, Private Banker). The journey doc has no separate "Analyst" or "Admin" archetypes — those are sub-roles inside any of the four. For this audit:

| User type (this audit) | Maps to journey doc | Modeled need |
|---|---|---|
| **CIO** | Persona A (Fund Manager) + B (Family Office) operator-side | Triage, decide, defend to LPs/IC. Wants density. |
| **Analyst** | Same workspace as CIO; lower trust | Edit positions, parse documents, draft briefings. Wants raw data + provenance. |
| **Admin** | Same workspace as CIO; superuser | Manage members, billing, audit, security. Wants control surfaces. |
| **Family Office (FO head)** | Persona B verbatim | Consolidation, IC narrative, liquidity certainty. Wants headline-quality reads. |
| **UHNW individual** | Persona C verbatim | "Is my money OK?" plain-English answer. Wants comprehension over precision. |

**Verification of journey doc against the live app:** The doc is roughly accurate but **two of its core promises are silently absent** in the live build:
1. The "consolidation across custodians" promise (job 1) is implemented as a single CSV import — the Sources/OAuth screen referenced in `FRONTEND_AUDIT.md` is not present in `frontend-mvp/`.
2. The "private markets" workflow promised to Persona B/C (PE NAV updates, capital calls, distributions) only exists at the document-extraction layer — no PE-specific surface exists in the nav.

The audit below treats those as **omissions per persona**, not just file inventory gaps.

## 1. Cross-cutting / platform issues

These break the experience for every persona and should be fixed before any per-page polish.

### 1.1 [P0] The default `AUTH_MODE=supabase` dev path is broken end-to-end

**What I observed:** `.env` ships with `AUTH_MODE=supabase` and live Supabase credentials. With that mode:
- `POST /api/auth/login` returns `200` with a `access_token` JWT in the body but **does not set `__crb_session` cookie**.
- The frontend stores the JWT in `localStorage` and sends `Authorization: Bearer <jwt>`, but `GET /api/auth/session` calls `_resolve_supabase_user`, which round-trips to Supabase's `/auth/v1/user`. With the locally-seeded user (different `auth_subject` from the issued JWT), this returns `401 Invalid session`.
- Net effect: **login succeeds, every subsequent page bounces to `/login.html?next=…`**. The app is unusable.
- Switching to `AUTH_MODE=local` works (legacy cookie path), which is how the rest of this audit was performed.

**Backend implications:** Either (a) provision the local DB user with the matching Supabase subject during seed, or (b) detect the JWT mismatch in `require_session` and auto-link by email, or (c) return a session cookie alongside the JWT in supabase mode so the cookie path keeps working. Recommend (b) — it survives Supabase resets without re-seeding.

**Files:** [backend/routers/auth.py:298](backend/routers/auth.py:298), [backend/services/auth/supabase.py:100](backend/services/auth/supabase.py:100).

### 1.2 [P0 — security] Secrets committed to `.env`

`.env` contains real-looking `SUPABASE_URL`, `SUPABASE_ANON_KEY`, `SUPABASE_SERVICE_ROLE_KEY`, and a `DATABASE_URL` with embedded password for `db.kyjsqehjxodwsrgmlezy.supabase.co`. Anyone with repo read access has admin Supabase access.

**Fix:** rotate all three keys + DB password immediately. Move secrets out of `.env` (use `.env.example` template + local untracked `.env`). Audit git history (`git log -- .env`) and consider that any historical commit may have leaked them — rotation is the only safe assumption.

### 1.3 [P0] Backend process instability (scheduler floods)

Backend logs show **2 jobs of the same name added per workspace per startup**, with the loop firing for every workspace row. After ~20s of normal operation the process became unresponsive in this audit and had to be restarted. Root cause: APScheduler `add_job` is being invoked inside a loop without job-id deduplication.

**Fix:** in `backend/services/scheduler.py`, pass a deterministic `id=` per (workspace, job-type) and `replace_existing=True`. Symptom-only fix is to filter the existing jobs before adding; correct fix is to register once at app startup and let the trigger handle workspaces.

### 1.4 [P1] Severity pills wrap mid-word

On the cockpit Active Risks register at 1440px desktop, `PRIORITY` / `ELEVATED` / `WATCH` pills render as `PRIOR\nITY`, `ELEVA\nTED`, `WATC\nH` — the assess column is too narrow for the pill width at the current font-size + uppercase + tracking. DESIGN.md specifies 999px radius, 10px font, 700 weight, 0.04em tracking — that spec is honored, but the cell width isn't enforced. `clientRects` reports a single line but the pill becomes 38px tall vs. 51px wide because flex-direction wraps the inline content visually. **The same pill renders correctly on the Briefing detail page** — so this is a cockpit-CSS-only regression.

**Fix:** in `frontend-mvp/_mvp.css`, give the assessment column a `min-width` ≥ pill width (or `white-space: nowrap` on `.mvp-pill`) and right-size the column.

### 1.5 [P1] Currency is inconsistent across pages of the same workspace

For Whitmore Family Office:
- Cockpit shows `CHF 7.7M` portfolio value.
- Assets shows `CHF 7,735,000`.
- Briefings list shows `Portfolio AUM of $8,500,000`.
- Briefing detail body shows `$8,500,000 · 1-day 95% VaR is 0.65% of AUM`.

Workspace `reporting_currency` is CHF (visible on Settings). The briefing generator is rendering its prompt template with USD as a hard-coded default. **For a Swiss family office, rendering AUM in USD on the IC-ready document is a credibility-destroyer.**

**Backend implication:** `briefing_runs.output_json` is being written with USD figures regardless of workspace currency. Either pass `workspace.reporting_currency` into the briefing prompt + serializer, or compute display currency at render time in [frontend-mvp/_app.js](frontend-mvp/_app.js) `formatCurrency()` and ignore whatever the model wrote. Prefer the former (numbers should match).

### 1.6 [P1] Liquidity outlook KPI contradicts its own subtext

Cockpit KPI strip shows:

> **LIQUIDITY OUTLOOK**
> None
> No capital call scheduled
> -CHF391.3K  Net 90-day liquidity
> [BUFFER BREACH chip] CHF1.7M shortfall to buffer

The same tile claims "None" (calls) AND a CHF 1.7M buffer breach. For an FO head or UHNW reader this reads as the system disagreeing with itself. The "None" headline is answering "are there capital calls" while the chip is answering "is the buffer healthy" — they're being collapsed into one tile.

**Fix:** split into two tiles ("Next capital call" + "Buffer status") OR change the headline metric to the buffer status (the more important read for a CIO) and demote calls to subtext.

### 1.7 [P1] Onboarding shows "5/5 complete" but still renders the upload UI

Visiting `/onboarding.html` after onboarding is finished shows a "Ready" banner *and* the Step 1/Step 2 file-drop UI underneath. A user who genuinely wants to re-import would have no clue if upload still works; a user who finished should never see this screen again.

**Fix:** if `state.is_complete`, redirect to `/cockpit.html` (the JS already has this branch at [frontend-mvp/_app.js:1038](frontend-mvp/_app.js:1038) for the *initial* mount but not for direct navigation). OR keep the page reachable but hide steps 1/2 behind a "Re-import portfolio" disclosure.

### 1.8 [P2] No global search / no command palette

Across 13 screens there is no quick way to jump to "AAPL" or "Carlyle Growth Fund III" or "Q1 statement". For an analyst, this means the table is the only way to find a position. For a CIO scanning before a meeting, it adds friction.

### 1.9 [P2] Sidebar has no Markets/Sources/Audit/Members nav items

`FRONTEND_AUDIT.md` lists Markets, Sources, Audit, and Members as Tier-1/Tier-2 demo screens. None are in the live left nav. The journey doc's "consolidation across custodians" job has no UI surface at all without a Sources screen.

## 2. Page-by-page findings

For each page: **what works**, **what's missing/broken**, then **per-persona impact**. Solutions consolidated in §3.

### 2.1 Login (`login.html`)

**Works:** Clean editorial layout, demo creds prefilled in placeholders, font system loads cleanly, mobile layout is fine.

**Issues:**
- [P2] No SSO buttons (Google, SAML) despite `FRONTEND_AUDIT.md` calling for them and the backend exposing the routes.
- [P2] No "Request access" link to public contact form.
- [P3] Right-rail "social proof stats" panel from the audit doc is not present (low value for MVP).

**Persona impact:** Admin needs SSO for enterprise sales; absence here means the enterprise conversation can't even begin.

### 2.2 Cockpit (`cockpit.html`)

**Works:** Editorial hero + section TOC right rail, KPI strip layout, donut composition with toggle (Asset Class / Sector / Geography), "Risk at a Glance" plain-English read with three bars (bad/severe/worst), "What is pushing downside now" tied to specific tickers.

**Issues:**
- [P1] Severity pills wrap (§1.4).
- [P1] Liquidity tile contradiction (§1.6).
- [P1] **Active Risks register has 35 entries with no triage / dedup / mitigation column.** "PRV_PE_FUND_A" appears in concentration, illiquidity, single-position, and single-name rows — same root issue, four entries. There's no "snooze / acknowledge / link to mitigation" surface, so the register is a printed list, not a working queue.
- [P2] "What is pushing downside" labels are jargon: `"Historical Scenario Additive estimate"`. Acceptable for analyst/CIO; opaque for FO/UHNW.
- [P2] Right-rail TOC (Composition / Risk at a Glance / Active risks) is sticky but redundant with the page hierarchy on a 1440px viewport — a CIO who scrolls once to the register won't return to the donut.
- [P3] No "as of" timestamp on the page header (only inside individual cards).

**Persona impact:**
- *CIO*: The 35-entry register is fine for triage in theory but unworkable without dedup/mitigation columns. Will print this and mark it up by hand — exactly the workflow the product is supposed to replace.
- *Analyst*: Loves the density. Wants drill-through (click a risk → see contributing positions, model output, agent reasoning). Currently the row is inert.
- *Admin*: Has no use for this page. Fine.
- *Family Office head*: Will not show this register to the family principal — it reads as a defect list, not a confidence statement. Needs an executive view above the register ("3 things to act on this week").
- *UHNW individual*: 35 risks at a glance is panic-inducing. Needs the page to lead with "Your money is mostly OK; here are the 2 things worth a conversation," not with 35 red dots.

### 2.3 Briefings list (`briefings.html`)

**Works:** Featured "Latest committee pack" card, archive list, "Generate briefing" CTA top-right, KPI tiles for Published / Drafts / Workflow.

**Issues:**

**Persona impact:**

- *Analyst*: No draft management surface — the "draft → review → publish" pipeline isn't real here.
- *Admin*: No workflow controls.
